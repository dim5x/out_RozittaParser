# Руководство для контрибьюторов

Rozitta Parser — проект в формате vibe coding. Любая помощь приветствуется, от тестирования до пул-реквестов.

## Как помочь

1. **Сообщить о баге** – создайте [Issue](https://github.com/Nynchezyabka/RozittaParser/issues) с подробным описанием (логи, шаги воспроизведения, версия ОС).
2. **Предложить исправление** – создайте Pull Request, опираясь на архитектурную документацию (`ARCHITECTURE.md`).
3. **Тестировать** – скачайте последнюю сборку из [Releases](https://github.com/Nynchezyabka/RozittaParser/releases) и проверьте работу на разных чатах/платформах.

## Сборка через GitHub Actions (рекомендованный способ)

Бинарные файлы для Windows, Linux и macOS собираются автоматически через GitHub Actions.

### Запустить сборку вручную

1. Перейдите на вкладку **Actions** репозитория.
2. Выберите workflow **Build desktop binaries** (или аналогичный).
3. Нажмите **Run workflow**.
4. При желании укажите тег для релиза (например, `v1.5.0`) и поставьте галочку **Publish to Releases**.
5. Через 5–10 минут бинарники появятся в артефактах или в разделе **Releases**.

### Автоматическая сборка

При пуше тега (например, `git tag v1.5.0 && git push origin v1.5.0`) workflow запускается автоматически и выкладывает бинарники в релиз.

### Локальная разработка (без сборки exe)

Для тестирования изменений достаточно запустить скрипт из исходников:

```bash
git clone https://github.com/Nynchezyabka/RozittaParser.git
cd RozittaParser
python -m venv .venv
.venv\Scripts\activate        # Windows
source .venv/bin/activate     # Linux/macOS
pip install -r requirements.txt
python main.py
```
⚠️ Важно: Локальная сборка через PyInstaller не поддерживается. Все бинарные релизы создаются через GitHub Actions.

### Кодстайл
Используем Python 3.11.

Следуем существующему стилю (примерно PEP 8, но с уже сложившимися отступами).

Где искать задачи
Открытые Issues с метками bug, enhancement, help wanted.

Три приоритетных бага описаны в ARCHITECTURE.md (фильтрация дат, большие видео, STT в exe).

Важно
Проект создан не профессиональным разработчиком, поэтому код может местами выглядеть странно. 

Все правки должны быть точечными и не ломать существующий функционал.

Если сомневаетесь – создайте Issue и обсудите.
